# StoreKit config for local testing

`Sagely.storekit` mirrors the App Store Connect "Sagely Membership" setup so you
can test the paywall (trials, upgrades, intro eligibility) **on the simulator**
without waiting for ASC approval or using a sandbox account.

- One subscription group, four products ranked to match ASC
  (`groupNumber` 1 = highest service):
  1. Max Annual · 2. Max Monthly · 3. Premium Annual · 4. Premium Monthly
- Every subscription carries a **7‑day free trial** (`P1W`, free) so you can
  verify "one trial per group" — once you start one, the others should drop the
  trial CTA and show "Upgrade".
- The two consumable Mochi top‑ups are included so product fetches don't 404.

Prices/levels here are for **testing only** — App Store Connect is the source of
truth for production.

## Enable it in Xcode

The iOS project is generated by `expo prebuild`, so `ios/` is gitignored — this
file lives outside it and must be re‑linked after any `expo prebuild --clean`.

1. Open `ios/<app>.xcworkspace` in Xcode.
2. **File → Add Files to "…"**, pick `mobile/storekit/Sagely.storekit`
   (uncheck "Copy items if needed" so it stays referenced here, not copied into
   the throwaway `ios/`).
3. **Product → Scheme → Edit Scheme… → Run → Options →
   StoreKit Configuration** → select `Sagely.storekit`.
4. Run on a simulator. Manage test purchases via
   **Debug → StoreKit → Manage Transactions…** (cancel/refund/reset to re‑test
   trial eligibility).

Verify in the editor that the **Subscription Rank** column reads 1→4 in the
order above (drag to reorder if not).

## RevenueCat note

RevenueCat builds its *offering* from the RC dashboard, so the dashboard still
needs these product IDs attached to the current offering for `getOfferings()` to
return them. With the StoreKit config active, the local file serves the product
metadata + handles purchases — no sandbox round‑trip.
